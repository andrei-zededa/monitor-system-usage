Analysis

Executive Summary

All 5 rate steps (10–100 kpps) completed with zero packet loss and zero kernel drops.
CPU utilization is dominated by QEMU/KVM overhead on host cpu8 and cpu9 (QEMU vCPU threads). At 100 kpps, cpu9 reaches 97% non-idle — primarily user+system+guest from KVM exit handling and virtio kick processing.
cpu9 is near saturation at 100 kpps. Higher packet rates will likely hit this bottleneck first, causing drops or throughput degradation.

Test Setup

UDP iperf test through a VM acting as a software bridge/forwarder. Traffic path:

The VM is far from its limit (91.5% idle aggregate). The bottleneck is on the host side: QEMU vCPU threads handling virtio kick/interrupt KVM exits consume ~180% of a single CPU across cpu8+cpu9.

Network — Zero Drops

No interface drops (rx_dropped, tx_dropped, rx_errors, tx_errors) on host or VM.
No softnet_stat drops or squeeze events on host or VM.
Host NET_RX softirq rate is negligible (~9/s) — all packet processing goes through the vhost-net/QEMU path, not the host's normal network stack.

Recommendations

Current capacity: The system handles 100 kpps without loss. cpu9 at 97% non-idle is the primary constraint. Estimated maximum throughput before drops: ~110–120 kpps.

To increase throughput capacity:

OptionImpactCommand / Config
Multi-queue virtio-net Distribute virtio processing across multiple vCPUs instead of concentrating on cpu8/cpu9 -device virtio-net-pci,mq=on,vectors=6,queues=2 in QEMU args
vhost-net thread pinning Ensure vhost threads run on optimal cores (same NUMA node as NIC) taskset -p <mask> <vhost-pid>
PCI passthrough / SR-IOV Bypass QEMU entirely; eliminate KVM exit overhead (50-80% reduction in host CPU per packet) Requires IOMMU + VF configuration
Increase VM vCPU count or enable RPS inside VM VM cpu1 softirq at 29% has headroom, but spreading across cores prevents future saturation Inside VM: echo f > /sys/class/net/enp0s2/queues/rx-0/rps_cpus

For this test specifically:

Extend rate steps to 120, 150, 200 kpps to find the actual throughput ceiling and confirm whether cpu9 saturation causes drops. The current test proves 100 kpps is within capacity.
