Analysis

Executive Summary

The throughput bottleneck is uneven virtio-net multi-queue IRQ distribution inside the VM. Despite 4 vCPUs and multi-queue vhost configured, the VM inconsistently activates RX queues across rate steps — using 1–2 cores at 100–160 kpps (causing 9–29% loss), all 4 cores at 220 kpps (0% loss), and only 3 cores at 280 kpps (33% loss). This variable queue activation explains the non-monotonic loss pattern.
Host CPU has ample headroom. Only 12 of 18 cores are used (cpu0-7 for vhost/networking, cpu8-11 for QEMU vCPUs). The host is not the bottleneck.

Network Interface Traffic

VM interfaces enp0s4 (tx) and enp0s5 (rx) show ~75.9k pps average over the test period (16.6M packets in ~219s). No interface-level drops were detected in sysfs counters. The drops occur at the softirq/iperf socket buffer level, not at the NIC ring buffer.

Root Cause Diagnosis

Variable virtio-net multi-queue activation. The 4 iperf flows use different source/dest IP pairs (10.99.2.200-203 → 10.99.1.100-103), which should hash to different virtio-net RX queues. However, the actual queue distribution is inconsistent:
  • At 100-160 kpps: only 1-2 of 4 queues receive traffic, saturating those vCPUs
  • At 220 kpps: all 4 queues are active, providing sufficient capacity
  • At 280 kpps: 3 of 4 queues are active, insufficient for the rate
This suggests the virtio-net RSS hash or irqbalance inside the VM is not distributing interrupts correctly across all configured queues.

Recommendations

1. Verify and Pin virtio-net Queue IRQ Affinity Inside VM

# Check number of queues
ethtool -l enp0s5

# Check current IRQ mapping
cat /proc/interrupts | grep virtio

# Pin each queue's IRQ to a specific vCPU
echo 1 > /proc/irq/<queue0_irq>/smp_affinity  # cpu0
echo 2 > /proc/irq/<queue1_irq>/smp_affinity  # cpu1
echo 4 > /proc/irq/<queue2_irq>/smp_affinity  # cpu2
echo 8 > /proc/irq/<queue3_irq>/smp_affinity  # cpu3

2. Disable irqbalance Inside VM

irqbalance may be moving IRQs between cores mid-test, causing the variable activation pattern. With manual pinning, disable irqbalance: systemctl stop irqbalance

3. Enable RPS as Fallback

# Distribute processing across all 4 vCPUs for each RX queue
echo f > /sys/class/net/enp0s5/queues/rx-0/rps_cpus
echo f > /sys/class/net/enp0s5/queues/rx-1/rps_cpus
echo f > /sys/class/net/enp0s5/queues/rx-2/rps_cpus
echo f > /sys/class/net/enp0s5/queues/rx-3/rps_cpus

4. Verify QEMU Multi-Queue Configuration

Ensure QEMU was launched with queues=4 on the virtio-net device and that the guest has activated all queues with ethtool -L enp0s5 combined 4.

5. Consider More vCPUs for Higher Rates

At 280 kpps, even with all 4 cores active at 220 kpps, each core was at 84-97% softirq. Scaling beyond ~250 kpps will require either more vCPUs or offloading with XDP/AF_XDP inside the VM.
