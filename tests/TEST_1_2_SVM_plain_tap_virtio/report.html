Analysis

Executive Summary

VM cpu1 is saturated at 96–97% softirq during the 220 and 280 kpps rate steps. All virtio NET_RX interrupt processing is concentrated on a single vCPU, capping forwarding throughput at ~191–194 kpps. The host has ample headroom (aggregate ~21% busy). No packet drops are detected at any interface; lost datagrams are dropped in the VM's forwarding path due to cpu1 being unable to drain the RX queue fast enough.

Root Cause: Single-Queue Virtio-Net Bottleneck

All NET_RX softirq processing inside the VM is pinned to cpu1. At 220 kpps, cpu1 reaches 96–97% softirq with only 3% idle. cpu0 assists partially (23–32% softirq) but cpu2 and cpu3 are completely idle. This is the classic single-queue NIC IRQ bottleneck — the virtio-net device has a single RX queue, so all incoming packets are processed by one CPU.

Network Counters

No rx_dropped or tx_dropped detected on any host or VM interface throughout the test.
No softnet_stat drops or squeeze events on the VM — the softirq budget is not being exhausted; the core simply can't process fast enough.
Host softnet_stat shows only 1 total change across the entire test window — effectively zero drops.

The packet loss reported by iperf (11.6% at 220 kpps, 31.8% at 280 kpps) occurs because the VM's bridge/forwarding path cannot keep up. Packets arriving through the virtio RX queue back up and are silently dropped when the queue fills — this is reflected in iperf's lost datagram count but not in interface-level counters since the loss happens before the packet reaches /proc/net/dev accounting in the forwarding direction.

Recommendations

1. Enable Multiqueue Virtio-Net (Primary Fix)

The single virtio-net RX queue is the root cause. Enabling multiple queues will distribute NET_RX processing across all 4 VM vCPUs, roughly 4x the throughput ceiling.

QEMU config change:

-device virtio-net-pci,netdev=net0,mq=on,vectors=10
-netdev tap,...,queues=4

Inside the VM:

ethtool -L enp0s2 combined 4

2. Enable RPS Inside VM (Quick Alternative)

If multiqueue is not available, RPS can distribute softirq processing across cores:

# Distribute to all 4 cores (bitmask 0xf)
echo f > /sys/class/net/enp0s2/queues/rx-0/rps_cpus
echo 4096 > /proc/sys/net/core/rps_sock_flow_entries
echo 1024 > /sys/class/net/enp0s2/queues/rx-0/rps_flow_cnt
RPS adds inter-processor interrupts (IPI overhead) — multiqueue is preferred as it avoids this overhead.

3. Consider PCI Passthrough / SR-IOV (Maximum Performance)

For production workloads requiring >500 kpps, consider bypassing the QEMU/vhost-net path entirely:

  • SR-IOV VF passthrough — assigns a Virtual Function directly to the VM, eliminating host-side overhead
  • vDPA — hardware-accelerated virtio data plane

These approaches eliminate the ~30% system time on host cores 4–7 (vhost threads) and the KVM exit overhead on cpu9.
