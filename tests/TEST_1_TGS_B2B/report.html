Analysis

Executive Summary

All rate steps (10–160 kpps) completed with 0% packet loss. The veth back-to-back path handles 160 kpps of 20-byte UDP datagrams with only ~10% aggregate CPU utilization across 18 cores. No CPU core exceeds 28% non-idle. No drops were observed at any layer (interface, softnet, or application).

This test establishes a baseline for the veth B2B data path without any bridge, QEMU, or tap overhead. The per-packet CPU cost is very low, confirming that the kernel network stack itself is not a bottleneck at these rates.

SoftIRQ Rate vs Load

NET_RX events/s tracks the packet rate almost exactly (e.g., 160,006/s at 160 kpps target). This is expected for veth: the transmitting CPU calls netif_rx() on the peer veth, triggering NET_RX softirq on the same CPU. NET_TX remains at zero because veth bypasses the TX softirq path entirely.

CPU Cost Breakdown

Client cores (0–3) carry more load than server cores because veth delivers packets in the sender’s CPU context. Each client core handles: user% (iperf sendmsg userspace) + system% (sendmsg syscall → UDP → IP → veth xmit → netif_rx) + softirq% (NET_RX for the receiving namespace, processed on the sending CPU).
Server cores (4–7) show only user% + system% with zero softirq%. Their work is purely the recvmsg syscall path (UDP socket receive). The NET_RX softirq for their packets already ran on the client cores.

Network Performance

Packet Drops

Zero drops at all layers:
  • iperf: 0% loss reported by all 4 server instances across all 6 rate steps
  • softnet_stat: drops = 0, squeeze = 0 throughout the entire test
  • sysfs interface counters: no changes on any physical interface (veth traffic stays within namespaces)

Interface Visibility

The veth interface counters (rx_bytes, tx_bytes, etc.) are not visible from the host namespace sysfs. The interface traffic charts show only enp86s0 and wlo1 (physical interfaces), which had no traffic during this test. To monitor veth traffic, run msu-collect inside the network namespaces or use /proc/net/dev from within the namespace.

Bottleneck Diagnosis

No bottleneck reached. At the peak rate of 160 kpps:
  • No core exceeds 28% non-idle — substantial headroom remains
  • No softirq budget exhaustion (squeeze = 0)
  • No backlog overflow (softnet drops = 0)
  • No iperf socket overflow (0% application-layer loss)

Based on the CPU scaling trend (approximately linear above 80 kpps), the per-core limit would be reached around 550–600 kpps per core at the current per-packet cost. With 4 client cores, the theoretical B2B veth throughput ceiling is approximately 2+ Mpps before any single core saturates.

Recommendations

This test serves as a clean baseline. To compare against bridged, TAP, or QEMU paths, run the same rate steps with traffic traversing those additional layers. The difference in CPU cost directly measures the overhead introduced by each layer.
