Analysis

Executive Summary

Throughput caps at ~184 kpps (received) at 280 kpps offered load due to VM vCPU softirq saturation. Three of four VM vCPUs reach 100% softirq processing the bridge forwarding path, while VM cpu0 sits idle — wasting 25% of available VM processing capacity. The multi-queue virtio-net configuration is distributing IRQs to only 3 of 4 vCPUs.
No kernel-level packet drops (rx_dropped, softnet drops) are observed on either host or VM. All packet loss occurs at the iperf UDP socket buffer level on the server side, where the iperf process cannot drain packets fast enough from a CPU already saturated with softirq.

Host cores 0–7 run vhost-net kernel threads (shown as system%) handling the virtio data plane. At 280 kpps they reach 17–33% system — well below saturation. No softirq appears on host cores because vhost-net processes packets entirely in kernel threads.

Host cores 9–11 run QEMU vCPU threads (guest% ≈ user% ≈ 38%, system% ≈ 7%). These are executing the VM’s softirq workload as guest code. The ~7% system is KVM exit handling (virtio kicks, interrupt injection). These cores are at ~83% non-idle — moderately loaded but not saturated.

Host cpu8 (QEMU vCPU 0) is idle, consistent with VM cpu0 being idle inside the guest.

Network Drop Analysis

No kernel-level drops detected on either host or VM.

Neither rx_dropped nor tx_dropped interface counters increase on VM interfaces (enp0s2, enp0s4, enp0s5). No softnet drops or squeezes are recorded in /proc/net/softnet_stat on either host or VM during traffic. All packet loss is at the iperf UDP application layer — the iperf server process cannot read from the socket fast enough when the vCPU it runs on is overloaded with softirq from the bridge forwarding path.

Root Cause Diagnosis

Primary bottleneck: VM softirq processing for bridge forwarding is limited to 3 of 4 vCPUs. At 280 kpps, these 3 cores are at 100% softirq, capping forwarding throughput at ~184 kpps.
Secondary issue: The multi-queue virtio-net setup is not distributing across all 4 queues/vCPUs. VM cpu0 remains idle during peak traffic, leaving 25% of forwarding capacity unused.
Observation: Loss distribution shifts between flow pairs across rate steps (flows 0+1 vs 2+3), suggesting IRQ affinity or queue mapping changes during the 30s idle gap between tests. This instability makes performance unpredictable.

Recommendations

1. Fix IRQ/Queue Distribution in VM (High Priority)

Ensure all 4 virtio queues are pinned to separate vCPUs inside the VM:

# Inside the VM — check current IRQ affinity
cat /proc/interrupts | grep virtio
# Pin each virtio queue IRQ to a separate CPU
echo 1 > /proc/irq/<irq_queue_0>/smp_affinity  # cpu0
echo 2 > /proc/irq/<irq_queue_1>/smp_affinity  # cpu1
echo 4 > /proc/irq/<irq_queue_2>/smp_affinity  # cpu2
echo 8 > /proc/irq/<irq_queue_3>/smp_affinity  # cpu3

Also disable irqbalance inside the VM to prevent it from moving IRQs between rate steps:

systemctl stop irqbalance

2. Verify Multi-Queue Configuration

# Confirm the number of queues matches vCPUs
ethtool -l enp0s4    # Check combined queue count
ethtool -l enp0s5
# If not already 4:
ethtool -L enp0s4 combined 4
ethtool -L enp0s5 combined 4

3. Enable RPS as Fallback

If IRQ affinity alone doesn’t spread to all 4 cores, use RPS to distribute softirq:

# Inside VM — enable RPS to all 4 CPUs
echo f > /sys/class/net/enp0s4/queues/rx-0/rps_cpus
echo f > /sys/class/net/enp0s5/queues/rx-0/rps_cpus
echo 4096 > /sys/class/net/enp0s4/queues/rx-0/rps_flow_cnt

4. Increase iperf Server Socket Buffer

To reduce application-level drops when softirq is high:

sysctl -w net.core.rmem_max=26214400
# Then restart iperf with larger buffer:
iperf -s -u -w 16M

5. Consider PCI Passthrough or SR-IOV

For maximum throughput, bypass the virtio/vhost stack entirely. PCI passthrough or SR-IOV removes the VM bridge forwarding overhead and the host vhost-net processing, potentially doubling throughput.
